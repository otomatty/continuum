# 製品要件定義書: Continuum

## 1. ビジョンと哲学

我々は、個人のキャリアが会社の枠組みを超える現代において、在籍中に得たスキルと実績が、個人の生涯にわたる「資産」となるべきだと信じる。Continuumは、その資産形成を全面的に支援し、オープンで健全な技術文化を内外に示す象徴となることを目指す。

## 2. プロジェクト概要

### 2.1 背景と目的
エンジニアのキャリアパスが多様化する中、企業は個人の成長を支援し、ポジティブな関係を長期的に築くことが求められる。本プロジェクトは、以下の目的を達成するために、会社のGitHub OrganizationにおけるOSS活動を可視化・促進するWebプラットフォームを構築する。

1.  **永続的な実績の構築:** 社員が個人アカウントで行ったコントリビューションを、退職後も消えることのない客観的な実績（ポートフォリオ）として形成する。
2.  **最先端技術によるスキルアップ:** フルスタックRustというモダンで挑戦的な技術スタックを採用し、本プラットフォームの開発自体を最高のスキルアップ機会とする。
3.  **オープンな文化の醸成:** 社内外からのコントリビューションを歓迎する文化を育み、「卒業生」とも技術で繋がる良好なアルムナイ関係を構築する。
4.  **ナレッジの集約と共有:** 組織内に散在する暗黙知を手軽に共有できる場を提供し、組織全体の技術力を底上げする。

### 2.2 スコープ
*   **対象:** 会社のGitHub Organization配下にある**パブリックリポジトリ**における全活動。
*   **対象外:** プライベートリポジトリの活動分析、GitHub以外のプラットフォーム連携、人事評価システムとの直接連携。

## 3. 利用者と役割

| 利用者 | 役割と期待 |
| :--- | :--- |
| **現役社員** | 自身の活動実績の確認、他者の活動からの学び、日々の知見の共有。スキルアップと実績作りの主役。 |
| **エンジニアリング<br>マネージャー** | 組織の技術活動の活性度や健全性を把握し、文化醸成の推進者として機能する。 |
| **元社員 (アルムナイ) /<br>外部コントリビューター** | 自身の過去の実績をポートフォリオとして参照。興味や愛着のあるプロジェクトに、所属の垣根を越えて継続的に貢献する。 |

## 4. 利用シナリオ

*   **現役社員Aさん:** 自身の個人ページを見て、在籍期間中のコントリビューショングラフを確認し、転職活動用の職務経歴書に実績として記載するURLを添付する。
*   **元社員Bさん:** 退職後に古巣のOSSで使われているライブラリに脆弱性を見つける。自身のGitHubアカウントからリポジトリをForkし、修正パッチのプルリクエストを送信する。
*   **マネージャーCさん:** ダッシュボードを眺め、最近外部（特に元社員）からのプルリクエストが増えていることを確認し、オープンな文化が機能していることを実感する。

## 5. 機能要件

### 5.1 ダッシュボード機能
*   組織全体の活動サマリー（コントリビューター数、リポジトリ数、外部からのPR数）を表示。
*   活動量に基づく週間/月間ランキングを表示。
*   リアルタイムのアクティビティタイムラインを表示。
*   リポジトリ一覧（Star数、最終更新日、コントリビューター構成比など）を表示。

### 5.2 知見共有機能
*   GitHub Discussionsをデータソースとした、ブログ形式のナレッジフィード。
*   カテゴリや投稿者による高度な検索・フィルタリング機能。
*   プラットフォーム内でのコンテンツ閲覧機能。

### 5.3 ポートフォリオ機能
*   **個人ページ:** ユーザーごとに専用ページを生成。そのユーザーの全コントリビューション（コミット、PR、レビュー等）を時系列やリポジトリ別に可視化する。このページは、社外にも共有可能な公開URLを持つ。
*   **組織貢献ハイライト:** 会社Organizationへの貢献に特化した活動履歴を表示する。

## 6. 非機能要件

| 項目 | 要件 |
| :--- | :--- |
| **パフォーマンス** | SSRによる高速な初期表示と、Wasmによるスムーズなクライアント動作を両立する。 |
| **セキュリティ** | GitHub OAuthによる認証。APIトークンはバックエンドで厳重に管理。 |
| **可用性** | Cloudflareのグローバルエッジネットワーク上で稼働し、高い可用性を確保する。 |
| **開発運用** | GitHubへのpushをトリガーとした自動テスト・デプロイ(CI/CD)。フロント/バックを単一リポジトリで管理するモノレポ構成。 |

## 7. システムアーキテクチャ

### 7.1 技術スタック
| コンポーネント | 技術 | 選定理由 |
| :--- | :--- | :--- |
| **統一言語** | **Rust** | パフォーマンス、信頼性、そして挑戦的なスキルアップ機会の提供。 |
| **バックエンド** | **Axum** | Tokioエコシステムとの完全な親和性とモジュラーな設計。 |
| **フロントエンド** | **Leptos** | Axumとのシームレスな統合(SSR)、最高のパフォーマンス。 |
| **実行基盤** | **Cloudflare** | サーバーレスによる運用負荷の低減とグローバルな高速配信。 |
| **データソース** | **GitHub API (GraphQL)** | 効率的なデータ取得と、実績の永続性を担保する唯一の真実のソース。 |

### 7.2 構成図
```
[ ユーザー (社員/元社員) ]
    ↓↑  (HTTPS)
[ Cloudflare Pages (Frontend: Leptos + Wasm) ]
    ↓↑  (API Call)
[ Cloudflare Workers (Backend: Axum on Rust + Wasm) ]
    ↓↑  (Cache R/W)
[ Cloudflare KV ]
    ↓↑  (GitHub API v4 GraphQL)
[ GitHub.com ]
```

## 8. 文化醸成のための運用指針

Continuumの成功は、ツールだけでなく、それを支える文化に依存する。以下の指針を掲げ、推進する。

*   **パブリック・バイ・デフォルトの原則:** 新規プロジェクトは、機密情報を含まない限り、原則としてパブリックリポジトリで開始することを推奨する。
*   **外部コントリビューションの歓迎:** 全てのパブリックリポジトリに`CONTRIBUTING.md`（貢献ガイドライン）を設置する。外部からのプルリクエストは迅速かつ丁寧にレビューする文化を醸成する。
*   **明確なライセンスポリシー:** 組織として推奨するOSSライセンス（例: MIT, Apache 2.0）を定め、全リポジトリに適用する。
*   **ポジティブな評価:** OSS活動への貢献を、日々の称賛（SlackでのShareなど）や、評価面談でのポジティブなトピックとして積極的に取り上げる。
